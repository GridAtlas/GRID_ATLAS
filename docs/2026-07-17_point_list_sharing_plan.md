# GRID ATLAS 地点リスト共有・リスト管理 実装計画書

作成日: 2026-07-17
対象: 地点リストを「ファイル」ではなく「名前付きリスト」として管理し、将来のネット共有の中核機能にするための実装計画

## 1. 目的

GRID ATLAS の共有価値を「地点単体の保存」ではなく、「誰かの視点で選ばれた地点リストを取り込み、必要なリストだけ格子上に重ねて見る体験」に移す。

想定例:

- ラーメン通が厳選した店リスト
- アニメファンが厳選した聖地一覧
- 旅行者が作った再訪候補地リスト
- 自分用の調査地点、候補地、訪問済み地点リスト

アプリ上ではファイル名を主役にしない。ファイルは共有やバックアップのための封筒であり、ユーザーが見る本体はリスト名を持つ地点リストとする。

## 2. 基本方針

### 2.1 用語

- 地点リスト: アプリ内で表示、非表示、保存、共有される地点のまとまり。
- リスト名: アプリ上でユーザーが認識する名前。例: `東京ラーメン30選`。
- ファイル: 地点リストを外部に渡すための入れ物。ユーザー体験上は副次的な存在。
- マイ地点: ユーザーが直接登録した地点を持つ既定リスト。

### 2.2 体験上の原則

- グリッドに表示するかどうかは、地点リスト単位のチェックで決める。
- インポート済みリストは削除しない限りアプリ内に残る。
- チェックを外してもリストは消えない。表示されなくなるだけ。
- ファイル名ではなく、リスト名を表示する。
- 旧形式の地点ファイル互換は切り、共有用の新形式に一本化する。
- 共有前に、地点数、リスト名、含まれる情報を確認できるようにする。

## 3. 現状との差分

現在の実装は `state.points` に全地点が直接入る。

```js
state.points = [];
```

地点リスト管理後は、内部的には `pointLists` を主データにする。

```js
state.pointLists = [
  {
    id: "local",
    name: "マイ地点",
    visible: true,
    editable: true,
    source: "local",
    points: []
  }
];
```

ただし、一度に完全移行すると影響範囲が大きい。段階的に移行する。

## 4. 推奨データモデル

### 4.1 Workspace v3

```json
{
  "version": 3,
  "projection": "web-mercator",
  "pointLists": [],
  "links": [],
  "loadedObservations": [],
  "viewport": {}
}
```

### 4.2 PointList

```json
{
  "id": "list-id",
  "name": "東京ラーメン30選",
  "description": "再訪したい店だけを集めたリスト",
  "author": "匿名",
  "visible": true,
  "editable": false,
  "source": "import",
  "importedAt": "2026-07-17T00:00:00.000Z",
  "createdAt": "2026-07-17T00:00:00.000Z",
  "updatedAt": "2026-07-17T00:00:00.000Z",
  "points": []
}
```

### 4.3 Point

既存 Point 形式を維持する。

```json
{
  "id": "point-id",
  "title": "店名または地点名",
  "note": "コメント",
  "photo": "",
  "photoName": "",
  "geo": { "lat": 35.0, "lng": 139.0 },
  "createdAt": "2026-07-17T00:00:00.000Z"
}
```

地点リスト共有では `note` と `photo` も含める。アプリはまだ開発段階でユーザーがいないため、下位互換よりも最良のデータ形を優先する。

## 5. 地点リスト共有ファイル形式

新形式は `grid-atlas-point-list` とする。既存の `grid-atlas-points` 互換は持たない。

```json
{
  "type": "grid-atlas-point-list",
  "version": 2,
  "exportedAt": "2026-07-17T00:00:00.000Z",
  "list": {
    "name": "東京ラーメン30選",
    "description": "再訪したい店だけを集めたリスト",
    "author": "匿名"
  },
  "points": []
}
```

複数リストを1ファイルにまとめる形式は将来拡張とする。

```json
{
  "type": "grid-atlas-point-list-bundle",
  "version": 2,
  "exportedAt": "2026-07-17T00:00:00.000Z",
  "lists": []
}
```

MVPでは単一リストファイルを優先する。

## 6. UI計画

### 6.1 スマホ

スマホの主画面は格子を最優先する。地点リスト管理は、将来的なハンバーガーメニュー内の `地点リスト` 画面に置く。

地点リスト画面のイメージ:

```text
地点リスト

☑ マイ地点                         12点
☑ 東京ラーメン30選                 30点
☐ 2026春 アニメ聖地巡礼            18点
☑ 旅行候補メモ                      9点

[読込] [保存] [共有]
```

操作:

- チェックオン: そのリストの地点をグリッドに表示
- チェックオフ: そのリストの地点を非表示
- リストタップ: リスト詳細、名前変更、削除、保存、共有
- 読込: ファイルまたは将来の共有リンクを地点リストとして追加
- 保存: 選択したリストをファイルとして書き出し
- 共有: 将来のオンライン共有に接続

### 6.2 PC

PCはサイドバーでリスト管理を常時確認できてもよい。スマホほど隠す必要はない。

推奨配置:

- データパネル内の `地点リスト` セクションをリスト一覧に置換
- 各リストにチェックボックス、点数、操作ボタンを配置
- 既存の分析パネルは、表示中リストだけを対象に集計

## 7. 描画と選択の扱い

### 7.1 表示対象

描画対象は `visible === true` の地点リストに含まれる地点だけ。

```js
function visiblePoints() {
  return state.pointLists
    .filter((list) => list.visible)
    .flatMap((list) => list.points.map((point) => ({ ...point, listId: list.id })));
}
```

既存コードの多くは `state.points` を直接参照しているため、段階的には `activePoints()` のような関数を導入し、参照箇所を置き換える。

### 7.2 選択

選択は `pointId` だけでは衝突の可能性がある。最終的には `listId + pointId` で識別する。

```js
{ type: "point", listId: "ramen-list", id: "point-id" }
```

ただし初期移行では、読み込み時に全地点IDをグローバルに一意化すれば、既存選択構造を延命できる。

### 7.3 線

判断が必要。候補は2つ。

- 案A: 線はワークスペース全体のローカル編集物として扱う。
- 案B: 線も地点リストに含める。

MVPでは案Aを推奨する。共有リストは地点だけに絞る方が分かりやすい。

表示ルール:

- 線の両端が表示中の地点なら表示
- 片方でも非表示なら線も非表示
- リストを削除した場合、そのリストの地点に関係する線は削除または無効化

## 8. インポート仕様

### 8.1 新形式読み込み

`grid-atlas-point-list` を読み込む。

処理:

1. JSONを検証
2. `list.name` を取得
3. name が空ならファイル名から推定、または入力させる
4. points を正規化
5. 既存IDと衝突する場合はIDを再採番
6. `state.pointLists` に追加
7. `visible: true` で表示
8. 保存
9. グリッドは必要に応じて全体表示

### 8.2 旧形式読み込み

旧形式は読み込まない。開発段階のため、互換層を増やさず `grid-atlas-point-list` v2 に統一する。

### 8.3 新規読込と追加読込の扱い

地点リスト管理に移行した後は、現在の `新規読込 / 追加読込` は次の意味に変える。

- 読込: リストとして追加
- 全リスト置換: 確認付きで既存地点リストを消して読み込みリストだけにする

通常操作としては `追加読込` を標準にし、`全リスト置換` は危険操作として奥に置く。

## 9. エクスポート仕様

### 9.1 保存対象

保存は `選択中の地点リスト` または `チェック中の地点リスト` を対象にする。

MVP推奨:

- リスト詳細画面から `このリストを保存`
- まず単一リスト保存だけ実装
- 複数リスト一括保存は将来拡張

### 9.2 共有対象から除外するもの

地点リスト保存では以下を除外する。

- 現在地
- 観察記録
- 線
- 端末由来の位置精度

写真とコメントは地点リストの魅力を支える情報として含める。ただし将来のオンライン共有では、共有前プレビューと注意喚起で誤共有リスクを抑える。

## 10. 移行計画

### Phase 1: 内部リスト管理の土台

目的: 見た目を大きく変えずに、内部だけ `pointLists` に寄せる。

実装:

- `state.pointLists` を追加
- 既存 `state.points` は `マイ地点` に移行
- `visiblePoints()` / `allPoints()` / `editablePoints()` を追加
- 描画、選択、検索、分析で `state.points` 直接参照を減らす
- 開発中のローカル地点があれば `マイ地点` に取り込む

受け入れ条件:

- 開発中のローカル地点を `マイ地点` に取り込める
- 既存の登録、選択、接続、削除、追跡が動く
- ローカル保存後に再起動しても状態が保たれる

### Phase 2: 地点リスト画面

目的: アプリ上でリスト名と表示チェックを管理できるようにする。

実装:

- データ画面に地点リスト一覧を追加
- チェックボックスで表示、非表示を切替
- リスト名、地点数、ソース種別を表示
- `マイ地点` は削除不可または確認強め
- 非表示リストの地点はグリッド、分析、選択対象から外す

受け入れ条件:

- チェックオフしたリストの地点が即座に消える
- チェックオンで再表示される
- リストを非表示にしてもデータは消えない

### Phase 3: インポートをリスト追加へ変更

目的: ファイル読込を「地点を足す」ではなく「リストを追加する」に変える。

実装:

- `grid-atlas-point-list` v2 読み込み対応
- 読み込み時にリスト名を決定
- 読み込み済みリストとして一覧に追加
- 既存の `新規読込 / 追加読込` UIを整理

受け入れ条件:

- 読み込んだ地点が名前付きリストとして並ぶ
- チェックで表示を切り替えられる

### Phase 4: リスト保存

目的: ファイル保存をリスト単位にする。

実装:

- リスト詳細から `保存`
- `grid-atlas-point-list` v2 形式で書き出し
- ファイル名は `grid-atlas-list-${safeListName}-${date}.json`
- アプリ上ではファイル名ではなくリスト名を維持

受け入れ条件:

- 保存したファイルを読み直すと同じリスト名で復元される
- 写真、コメントの扱いが仕様通りになる

### Phase 5: 共有前安全確認

目的: 自宅や職場などの誤共有リスクを下げる。

実装:

- 共有前プレビュー
- 地点数、リスト名、含まれる情報を表示
- コメントと写真が含まれることを明示
- `現在地は共有されません` を明示

受け入れ条件:

- 共有前に内容を確認できる
- コメント、写真、現在地の扱いが誤解されにくい

### Phase 6: オンライン共有への接続

目的: ファイルを直接渡さなくても、リストURLで取り込めるようにする。

実装候補:

- Cloudflare Workers + R2 + D1
- 共有ID発行
- 期限付きリンク
- 暗号化リンク型の検討
- QRコード表示

この段階は別計画書のオンライン共有ロードマップと接続する。

## 11. 既存機能への影響

### 11.1 登録

登録先は原則 `マイ地点`。将来的には登録先リストを選べるようにしてもよい。

### 11.2 編集

初期は `マイ地点` のみ編集可能にする。インポート済み共有リストは編集不可または複製して編集。

### 11.3 削除

- 地点削除: マイ地点では削除可能
- 共有リスト内地点: 初期は不可
- リスト削除: インポート済みリスト単位で削除可能

### 11.4 接続

表示中の地点同士なら接続できる。非表示リストの地点に接続された線は表示しない。

### 11.5 巡回、追跡、対象、起点

表示中の地点だけを対象にする。対象や起点に指定された地点がリスト非表示になった場合は、指定解除または一時無効化する。

## 12. 判断が必要な事項

### Q1. インポート済み共有リストは編集不可でよいか

推奨: 編集不可。編集したい場合は `マイ地点にコピー` または `複製して編集`。

理由: 共有元のリストと自分の編集が混ざると、再共有や更新の意味が曖昧になるため。

### Q2. コメントを共有対象に含めるか

決定: 含める。

理由: 共有リストの価値は、地点名だけでなく選定理由や補足情報にもあるため。

### Q3. 写真を共有対象に含めるか

決定: 含める。

理由: 現状は開発段階であり、リスト体験として自然な完全性を優先する。オンライン共有時には容量、プライバシー、著作権、位置情報メタデータの扱いを別途安全設計に入れる。

### Q4. 線も地点リストに含めるか

推奨: MVPでは含めない。

理由: まずは地点リスト共有の価値を明確にする。線はユーザーが自分のワークスペースで作る分析結果と考える。

### Q5. リスト表示チェックを外した時、関連する線や選択はどうするか

推奨: 線は非表示、選択は解除、起点/対象は一時解除。

理由: 画面に見えない地点が操作対象として残ると直感に反する。

### Q6. `マイ地点` は複数リスト化するか

推奨: 初期は `マイ地点` 1つ。あとから `自分のリストを作成` に拡張。

理由: いきなり複数の編集可能リストにすると登録先の概念が増えすぎる。

### Q7. 旧ファイル読み込み時にリスト名をどう決めるか

決定: 旧ファイルは読み込まない。

理由: まだ開発段階でユーザーがいないため、下位互換よりもデータ体系の単純さを優先する。

### Q8. 保存ボタンは何を保存するか

推奨: 選択した単一リストを保存。複数一括保存は後回し。

理由: ユーザーにとって「このリストを送る」が一番理解しやすい。

## 13. 推奨する最初の実装単位

最初に実装するなら、以下の小さな塊がよい。

1. `state.pointLists` を追加し、既存地点を `マイ地点` に移行する。
2. `visiblePoints()` を追加し、描画と分析を表示中地点ベースにする。
3. データ画面に地点リスト一覧を出す。
4. `マイ地点` のチェック切替だけを実装する。
5. 既存の地点読込を、名前付きリストとして追加するように変更する。

ここまでで「地点リストが並んで、チェックで表示を切り替える」という核が完成する。

## 14. 実装時の注意

- 外部ファイル互換は持たず、現行開発データだけを必要最小限で `マイ地点` に取り込む。
- workspace は v3 の `pointLists` を主データにする。
- `state.points` 直接参照は一気に消すより、`visiblePoints()` 経由に段階移行する。
- ID衝突は読み込み時に必ず解消する。
- 非表示リストの地点が選択や起点、対象に残らないようにする。
- 共有UIでは、ファイル名よりリスト名を常に優先する。
